# Testing Rules

> **ОБРАЗЕЦ — отредактируйте под практики вашей команды.** Это шаблон с
> разумными значениями по умолчанию для JUnit 5 + Kafka/async. Замените примеры
> своими правилами; удалите неприменимое.
>
> Этот файл **инжектится в контекст агента** на старте сессии и при запуске
> сабагента `coder` (механизм — `gate_context_inject`, список в `RULE_FILES`).
> Держите его компактным.

## Framework и структура

- Тесты на JUnit 5 (`org.junit.jupiter`). Ассерты — AssertJ/JUnit assertions,
  без «голых» `assert`.
- Один тест проверяет одно поведение; структура Arrange–Act–Assert
  (given/when/then).
- Имя теста описывает поведение и условие: `returnsFinalStatus_whenProviderTimesOut`.
- Тесты детерминированы и независимы от порядка запуска; без общего
  изменяемого состояния между тестами.

## Асинхронность — без `Thread.sleep`

- **Никаких `Thread.sleep` для синхронизации** — это flaky. Для ожидания
  условия использовать **Awaitility** (`await().atMost(...).until {...}`).
  `Thread.sleep` в тестовых файлах ловит `gate_clean_code` (advisory).
- Корутины тестировать через `runTest` (`kotlinx-coroutines-test`); у каждого
  ожидания — конечный таймаут, чтобы зависший тест падал, а не висел.
- У интеграционных ожиданий всегда есть верхняя граница времени.

## Kafka и интеграционные тесты

- Интеграционные Kafka-тесты — на **Testcontainers** (реальный брокер в
  контейнере), не на самодельных моках брокера и не на `EmbeddedKafka` без
  необходимости.
- Проверять консьюмера через фактическую публикацию в топик и ожидание эффекта
  (Awaitility), а не через прямой вызов метода-обработчика в обход Kafka.
- Покрывать политику ошибок: retry, DLQ, идемпотентность при повторной доставке.

## Объём и приоритеты

- Покрытие пропорционально риску: путь оплаты/деньги/внешние вызовы — тесты
  обязательны; тривиальные геттеры — нет.
- На каждый багфикс — регрессионный тест, который падал бы до фикса
  (сначала воспроизвести провал, затем чинить).
- Не подгонять тест под текущую (возможно неверную) реализацию: тест выражает
  ожидаемое поведение из спеки/критериев приёмки.

## Чего избегать

- Тестов, завязанных на приватные детали реализации вместо наблюдаемого поведения.
- Недетерминизма: реальное время/`now()`, сеть без контейнера, случайные порты
  без изоляции, зависимость от системной локали/таймзоны.
- «Зелёных» тестов без ассертов (проверка, что «не упало», — не тест).

# Реверс-анализ в GigaCode — руководство аналитика

Это подробное и по возможности понятное описание того, как устроен и как
работает модуль аналитики GigaCode. README рядом — краткий вход; здесь — полный
разбор: с чего начать, какой софт нужен, как идёт процесс и как он устроен внутри.

Документ рассчитан и на аналитика без сильного технического бэкграунда: все
термины объясняются простыми словами в разделе [Словарь](#словарь).

---

## Коротко: что это и зачем

**Реверс-анализ** — это разовая «фотография» одной бизнес-функции по уже
существующему коду. Вы берёте фичу (например, «Блокировка карты»), а GigaCode
помогает превратить код в документацию: что функция делает, как устроен её
поток, с чем интегрируется, какие данные использует — с явными пометками, что
подтверждено кодом, а что пока предположение.

Ключевая идея: **это bootstrap (первичная загрузка), а не постоянное ведение
документации.** Вы один раз создаёте базу по фиче. Дальше документация живёт уже
своей жизнью (через отдельный процесс изменений — он за рамками этого модуля).

Результат — **двухслойный**:

1. **Технический слой** — рабочие черновики на русском (AsciiDoc) в
   `docs/features/<фича>/`. После прогона «замораживается».
2. **Финальный слой** — корпоративное дерево документации (`analytics/`,
   `architecture/`) плюс спецификация в `openspec/`.

## Быстрый старт

### Шаг 1. Поставьте обязательное

Нужны только три вещи (см. [Требуемый софт](#требуемый-софт)): GigaCode CLI,
Git и Python 3. Всё остальное — опционально.

### Шаг 2. Проверьте, что модуль рабочий

Из каталога модуля запустите smoke-проверку (она полностью офлайн — не требует
ни сети, ни MCP, ни GigaCode):

Windows (PowerShell):

```powershell
cd <repo>\modules\analytics
.\scripts\smoke-check.ps1
```

Linux / macOS (Bash):

```bash
cd <repo>/modules/analytics
bash scripts/smoke-check.sh
```

Ожидаемый финал — строка `Analytics module smoke check passed.` Если она есть,
конфигурация цела.

### Шаг 3. Запустите GigaCode и реверс-анализ

```text
gigacode
```

Внутри GigaCode дайте команду — **одна фича за прогон**, обязательно укажите
название и контекст источников:

```text
/reverse-analysis feature "Блокировка карты" jira ABC-123
```

Если Jira/Confluence не используются — скажите это явно, иначе система
переспросит:

```text
/reverse-analysis feature "Блокировка карты" code-only, без Jira, без Confluence
```

### Шаг 4. Пройдите процесс

GigaCode проведёт вас по 9 шагам (см. [Как идёт процесс](#как-идёт-процесс)).
Важный момент: **после построения карты кода система остановится и спросит вас
подтвердить область анализа.** Это единственная обязательная «ручная»
остановка — проверьте, что в анализ попало то, что нужно, и подтвердите.

В конце вы получите замороженные техдоки, спецификацию и финальное дерево
документации (см. [Что получится на выходе](#что-получится-на-выходе)).

## Требуемый софт

### Обязательное

| Софт | Зачем | Проверка |
|---|---|---|
| **GigaCode CLI** | сам агент; команда `gigacode` | `gigacode --version` |
| **Git** | модуль работает в git-репозитории | `git --version` |
| **Python 3** | на нём написаны хуки/гейты и smoke-проверки | `python --version` |
| **PowerShell** (Windows) / **Bash** (Linux/macOS) | запуск smoke-проверок | — |

Этого достаточно, чтобы пройти весь процесс. Опциональные инструменты ниже
лишь ускоряют и обогащают работу — без них всё функционирует с фолбэками.

### Опциональное (у каждого есть фолбэк)

| Софт | Что даёт | Как поставить | Если нет |
|---|---|---|---|
| **Serena MCP** | семантический поиск по коду (находит символы «по смыслу», а не по тексту) | `pip install uv` → `uv tool install -p 3.13 serena-agent` → `serena init` | агенты ищут через `rg` (текстовый поиск) — медленнее, но работает |
| **Graphify** | карта модулей репозитория (граф зависимостей) | `pip install graphifyy`, затем `/graphify .` и `python scripts/build_module_map.py` | карта не строится — система молча работает без неё; можно вести карту вручную в `journal.md` |
| **Context7 MCP** | свежая документация библиотек | по умолчанию **выключен**; чтобы включить — добавьте сниппет в `settings.json` (см. README), нужен Node.js 18+ | анализ идёт на знаниях модели и официальной документации |
| **Atlassian MCP** | доступ к Jira / Confluence | настраивает ваша команда | анализ только по коду и вашему вводу; в результате явно фиксируется, что внешний контекст не использовался |

Важно: **репозиторий не устанавливает MCP-серверы и не хранит никаких ключей и
токенов.** Любой опциональный инструмент вы ставите сами, на своей машине.

## Что получится на выходе

Один прогон оставляет три согласованных результата. Источник истины —
**код**; всё выводится из него по цепочке.

```text
        КОД  (источник истины о текущей реализации)
         │
         ▼
1) ТЕХНИЧЕСКИЙ СЛОЙ — docs/features/<фича>/
     overview.adoc       — что за фича, обзор
     flow.adoc           — основной сценарий, ветвления, ошибки
     integrations.adoc   — API, очереди, события, внешние системы
     data.adoc           — сущности, таблицы, идентификаторы
     questions.adoc      — открытые вопросы, предположения, противоречия
     journal.md          — рабочий журнал (область, карта кода, решения)
     manifest.json       — машиночитаемый «паспорт» прогона
         │   (после проверки доказуемости — «заморожен»)
         ▼
2) СПЕКА — openspec/specs/<capability>/spec.md
     функциональные требования как «текущая истина»
         │
         ▼
3) ФИНАЛЬНЫЙ СЛОЙ — analytics/ + architecture/
     корпоративное дерево по типам (см. ниже)
```

### Метки доказуемости

В техническом слое каждое утверждение помечается источником и статусом — чтобы
читатель видел, чему доверять:

- `Источник: код` / `jira` / `confluence` / `пользователь`
- `Статус: предположение` / `открытый вопрос`

Подтверждённые факты отделены от догадок. Файл `questions.adoc` существует
специально, чтобы открытые вопросы не растворялись в тексте.

### Финальное дерево (корпоративный стандарт)

```text
architecture/                       # диаграммы PlantUML (C4, последовательности), .puml
analytics/
├── db/
│   ├── ddl/                        # скрипты схемы (не заполняется при bootstrap)
│   ├── dml/                        # скрипты данных (не заполняется при bootstrap)
│   └── data-model/                 # модель существующей схемы, .dbml
├── functional-requirements/        # .adoc, производное от спеки
├── integration/                    # ВНЕШНИЕ интеграции
│   ├── event/                      # AsyncAPI .yaml + схемы событий .json
│   ├── rest/                       # OpenAPI .yaml
│   ├── mapping/                    # маппинг сервис↔БД, .adoc
│   └── nfr and contact/            # NFR и контакты соседних команд, .adoc
├── api/                            # ВНУТРЕННИЕ API
│   ├── event/                      #
│   ├── rest/public/ , rest/private/
│   ├── mapping/
│   └── nfr/
├── use-case/                       # сценарии использования, .adoc
└── glossary/                       # глоссарий, .adoc
```

Правила именования (их проверяет автоматика): **файлы — `UpperCamelCase`**
(например, `BlokirovkaKarty.adoc`), **каталоги — `kebab-case`** (два каталога
из стандарта пишутся как есть: `nfr and contact` и `data-model`). Содержимое —
на русском.

## Как идёт процесс

Процесс — это 9 шагов. За прогон в файле `manifest.json` меняется статус:
`scoping → draft → confirmed → complete`. Статус — это «где мы сейчас»; на него
опираются автоматические проверки.

```text
[1] Запуск /reverse-analysis            preflight проверяет полноту запроса
        │
[2] Intake (в основной сессии)          вы описываете область → journal.md
        │                               создаётся manifest.json (статус: scoping)
        ▼
[3] Карта кода  (агент code-mapping)    точки входа, потоки, интеграции, данные
        │                               записывается в journal.md
        ▼
   ⛔ ОСТАНОВКА: вы подтверждаете область анализа
        │
[4] Черновик техдоков (агент documentation)   5 файлов .adoc   (статус: draft)
        │
[5] Проверка доказуемости (агент verifier)    нет ли голословных утверждений,
        │                               противоречий, пропущенных меток
        │                               правки вносятся в техслой (статус: confirmed)
        ▼
[6] Извлечение спеки (агент documentation)    openspec/specs/<capability>/spec.md
        │                               (пишется один раз на новую capability)
        ▼
[7] Финальная генерация (агент documentation) analytics/ + architecture/
        │
[8] Проверка деривации (агент verifier)       финал ↔ спека ↔ техслой согласованы?
        │
[9] Закрытие                            manifest.json: статус complete,
                                        техслой замораживается
```

### Почему так

- **Intake — в основной сессии, не в отдельном агенте.** Задавать вам вопросы
  может только основной диалог, поэтому отдельного «агента опроса» нет.
- **Сначала карта, потом текст.** Нельзя описывать поведение, не найдя его в
  коде. Карта кода — это фундамент.
- **Остановка для подтверждения области.** Чтобы не потратить время на анализ
  не того. Это ваша точка контроля.
- **Дисциплина деривации: код → техдоки → спека → финал.** Любая найденная
  позже ошибка чинится наверху цепочки (в техдоках), а производные артефакты
  перегенерируются. Финал и спеку не правят «на месте» — иначе слои разойдутся.
- **Проверка доказуемости идёт ДО генерации финала.** Поэтому к концу прогона
  техслой и финальное дерево совпадают по построению.

### Три агента

| Агент | Что делает | Права |
|---|---|---|
| `code-mapping` | строит карту кода по подтверждённой области (Serena → `rg`, карта модулей) | в основном чтение |
| `documentation` | пишет техдоки → спеку → финальное дерево | пишет в `docs/features/`, `openspec/specs/`, `analytics/`, `architecture/` |
| `verifier` | проверка доказуемости (шаг 5) и проверка деривации (шаг 8) | только чтение, выдаёт замечания |

## Архитектура (как это устроено внутри)

Вам как аналитику не нужно знать внутренности, чтобы работать. Но понимание,
«почему GigaCode иногда что-то блокирует», экономит нервы. Здесь — по-простому.

### Идея: правила соблюдаются автоматически, а не «на честном слове»

Модуль не полагается на то, что агент «вспомнит» правила. В ключевые моменты
GigaCode сам запускает проверки (**хуки** и **гейты**), которые могут
**разрешить**, **предупредить** или **заблокировать** действие. Это и есть
«enforcement» — принуждение к стандарту.

> Защита **многослойная**, но не абсолютная: гейт `git_guard` разбирает
> произвольные shell-команды, а это принципиально дырявая задача. Что именно
> гарантируется, а что остаётся за слоем `settings.json` + ревью человеком —
> честно описано в [`enforcement-limitations.md`](enforcement-limitations.md).

### Один вход: роутер

Все хуки идут через единственную точку — `router.py`. GigaCode на каждое событие
зовёт роутер, а тот по таблице `router.config.json` решает, какие проверки
запустить. Каждое решение записывается в журнал `.gigacode/logs/decisions.jsonl`
(удобно потом понять, почему что-то заблокировалось).

```text
  событие GigaCode                    router.py            гейты
  (старт сессии,      ──────►   читает router.config  ──►  preflight_check
   ввод запроса,                 и запускает нужные         gate_context_inject
   правка файла,                 гейты по очереди           git_guard
   git-команда,                                             gate_spec_bootstrap
   конец прогона)     ◄──────   один ответ:           ◄──  gate_techdocs
                                allow / warn / block        gate_final_format
                                                            validate_run_output
                                       │
                                       ▼
                          .gigacode/logs/decisions.jsonl  (журнал решений)
```

### Какие проверки и когда

| Когда (событие) | Проверка | Что делает |
|---|---|---|
| вы отправили запрос | `preflight_check` | если это реверс-анализ, требует название фичи и контекст источников |
| старт сессии / старт агента | `gate_context_inject` | подсовывает агенту правила и карту модулей (если есть) |
| перед git-командой и правкой файла | `git_guard` | блокирует опасные операции (напр. в защищённые пути) |
| перед записью спеки | `gate_spec_bootstrap` | разрешает создать спеку capability только один раз; правка существующей — через отдельный процесс |
| после записи техдока | `gate_techdocs` | проверяет: это AsciiDoc, на русском, есть обязательные атрибуты заголовка |
| после записи финального файла | `gate_final_format` | проверяет имена/расположение по корпоративному стандарту |
| в конце прогона | `validate_run_output` | по `manifest.json` проверяет, что прогон реально завершён (нужные файлы на месте) |

«Предупреждение» (warn) не останавливает работу — это совет. «Блок» (block)
останавливает действие с понятным сообщением о причине.

### Контекст-провайдеры (опционально)

- **Serena** — даёт агентам семантический поиск (`find_symbol`). Подключена в
  `settings.json` как MCP-сервер; напоминания работают асинхронно — если Serena
  не установлена, ничего не тормозит.
- **Graphify → карта модулей.** Скрипт `scripts/build_module_map.py` превращает
  граф Graphify в компактный `.gigacode/context/module-map.md`, который
  `gate_context_inject` подкладывает агентам на старте. Нет файла — секция
  просто пропускается.
- **Context7** — свежая документация библиотек; по умолчанию выключен.

### Если хуки мешают

Аварийный выключатель: в `.gigacode/settings.json` установите
`"disableAllHooks": true` — все проверки отключатся. Это временная мера для
отладки, а не штатный режим.

### Спецификации (OpenSpec)

Реверс-анализ использует только «половину спецификаций» OpenSpec: спека
пишется как **результат** (текущая истина по capability), без процесса
изменений. Структурные заголовки в спеке — на английском (этого требует
валидатор: `### Requirement:`, `SHALL`, `#### Scenario:`, `WHEN/THEN`), а текст
требований — на русском. Правила формата — в `rules/openspec.md`.

## Работа офлайн и фолбэки

Модуль спроектирован так, чтобы работать без сети и без опциональных
инструментов:

- Нет Serena → агенты ищут через `rg`.
- Нет Graphify/карты → гейт молча пропускает секцию, карту можно вести вручную.
- Нет Context7 → анализ на знаниях модели и официальной документации.
- Нет Atlassian → анализ по коду и вашему вводу, ограничение фиксируется в
  результате.
- smoke-проверки проходят полностью офлайн и не требуют ничего из опционального.

## Проверка под вашу сборку GigaCode

GigaCode — форк Qwen Code. Имена событий и инструментов в `router.config.json`
взяты из документации Qwen Code. Перед продакшеном стоит один раз свериться:
временно подключите `python .gigacode/hooks/hook_probe.py` на интересующие
события в `settings.json`, выполните типовые действия (запрос, правка файла,
git-команда) и сравните, что записалось в `.gigacode/logs/hook-probe.jsonl`, с
матчерами в `router.config.json`.

## Словарь

- **Хук (hook)** — программа, которую GigaCode сам запускает в определённый
  момент (старт сессии, перед записью файла, перед git-командой, в конце). Вы
  её не вызываете.
- **Гейт (gate)** — отдельная проверка внутри процесса. Может разрешить,
  предупредить или заблокировать действие.
- **Роутер (router)** — единая точка входа для всех хуков; по таблице решает,
  какие гейты запустить.
- **Агент (субагент)** — специализированный помощник GigaCode с узкой задачей
  (карта кода / документация / проверка).
- **MCP** — протокол, через который GigaCode подключает внешние инструменты
  (Serena, Context7).
- **OpenSpec** — формат спецификаций. Здесь используется только для записи
  спеки как текущей истины.
- **Capability** — связная способность/функция системы, по которой пишется
  одна спека (часто совпадает с анализируемой фичей).
- **Спека (spec)** — файл `openspec/specs/<capability>/spec.md` с
  функциональными требованиями; «текущая истина».
- **Деривация** — вывод одного артефакта из другого по цепочке
  код → техдоки → спека → финал.
- **AsciiDoc** — текстовый формат разметки для документации (как Markdown, но
  богаче); финальные документы и техслой пишутся на нём.
- **manifest.json** — «паспорт» прогона: фича, дата, коммит кода, статус и
  список произведённых файлов.
- **bootstrap** — первичная разовая загрузка документации по существующему коду
  (то, что делает реверс-анализ).
- **Метки доказуемости** — пометки `Источник: …` и `Статус: …`, отделяющие
  факты от предположений.

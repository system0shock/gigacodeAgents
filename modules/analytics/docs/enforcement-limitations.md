# Модель защиты и остаточные риски

Этот документ честно описывает, **что enforcement-слой модуля гарантирует, а что
— нет**. Он нужен мейнтейнерам и тем, кто решает, можно ли полагаться на гейты в
конкретной сборке GigaCode. Аналитику для работы он не требуется — см.
[`flow-overview.md`](flow-overview.md).

Короткий вывод: защита **многослойная и фейл-клоузед в основном**, но один из
слоёв (`git_guard`) разбирает **произвольную shell-команду**, а это принципиально
дырявая задача. Поэтому `git_guard` — не единственная и не последняя линия: его
подстраховывают структурный backstop, гейты файловых инструментов и слой
`settings.json` + ревью человеком.

## Слои защиты (defense-in-depth)

1. **Гейты файловых инструментов** (`WriteFile`/`Edit`/`NotebookEdit`, событие
   `PreToolUse`). Запись в `.gigacode/**` и `.git/**` — **BLOCK**; в
   `openspec/specs/**` — правило create-once (`gate_spec_bootstrap`); финальное
   дерево — формат/размещение (`gate_final_format`). Эти пути GigaCode передаёт
   как явные аргументы, поэтому слой **детерминированный и надёжный**.
2. **`git_guard` на shell-канале** (`Bash`/`Shell`, `PreToolUse`). Блокирует
   деструктивный git и запись/удаление защищённых путей через шелл. Это слой над
   **недоверенным открытым вводом** — см. ниже про границы.
3. **Структурный backstop** (`_self_protect_catch_all`). Любая не-read-only,
   не-git команда, которая **просто упоминает** `.gigacode`/`.git`/openspec-путь
   в аргументах, блокируется (read-only `cat`/`grep`/`ls`/… разрешены по
   allow-list). Это и есть страховка на случай, когда разбор «какая программа
   запускается» промахнулся: класс обхода сужается с «любая фича shell» до
   «эффект полностью скрыт от текста аргументов».
4. **Stop-гейт** (`validate_run_output`). Прогон со статусом `complete` обязан
   иметь реальные финальные файлы (не пусто/`.gitkeep`/каталог), техдоки и спеку.
5. **`settings.json` (allow/deny на инструменты) + ревью человеком.** Это
   **внешний** по отношению к парсингу слой; именно он закрывает остаток ниже.

## Что `git_guard` надёжно блокирует

Закрыто и покрыто офлайн-тестами (`scripts/test_gates.py`) + adversarial-прогоном:

- `git reset --hard`, `git clean -f` (без `-n`), force-push и delete/`+`/`:`
  refspec, `git branch -d/-D/--force`, `git checkout`/`switch`/`restore`,
  `git stash clear/drop`, `worktree remove`, `reflog expire/delete/drop`,
  `gc --prune`, `update-ref`/`filter-branch`/`filter-repo`.
- `git rm`/`git mv` по защищённым путям; `git checkout <файл>`/`.`/pathspec-магия
  (`:/…`, `:(top)…`).
- Запись/удаление `.gigacode`/`.git`/`openspec` через шелл: редиректы
  (`>`,`>>`,`n>`,`>&`,`x>file`, ведущие/срединные), `cp`/`mv`/`tee`/`install`/
  `-t DIR`, `dd of=`, `sed -i`, `touch`/`mkdir`/`ln`/`chmod`/`tar`/`rsync`/
  `New-Item` и пр. (через структурный backstop), `find -delete`/`-exec…`.
- Обходы разбора: обёртки (`sudo`/`env`/`nice`/`timeout`/`xargs`/… c value-опциями
  и длительностями), `bash -c`, `eval`, подстановки `$()`/`` `` ``/`<()`,
  control-блоки (`if … then …; { …; }`), **тела функций** (`f(){ … }; f`),
  inline-алиасы (`-c alias.*`, `--config-env`, `GIT_CONFIG_*`),
  unicode/NBSP-разделители, абсолютные/`..`-пути, `git.exe`/`\git`.

## Остаточные риски (что НЕ ловится)

`git_guard` инспектирует **текст команды**. Принципиально вне его охвата —
эффекты, **скрытые от текста аргументов**:

1. **Раскрытие переменных окружения в саму команду.** `c='git reset --hard'; $c`
   или `eval "$VAR"` со значением из окружения — токены команды не видны гейту.
2. **Here-doc / здесь-строки и stdin-питание.** `xargs rm < list` или
   `cmd <<'EOF' … EOF` — удаляемые/исполняемые данные приходят из stdin, а не из
   аргументов.
3. **Скомпилированные/сторонние хелперы.** `python -c "..."`, скрипт, бинарь —
   если защищённый путь не упомянут в аргументах, backstop его не увидит
   (если упомянут — блок).
4. **Недокументированные value-опции обёрток.** Таблица value-опций
   (`timeout`/`xargs`/…) покрывает документированные; экзотический неизвестный
   флаг со значением может сдвинуть определение команды.
5. **Семантика рантайма GigaCode не проверена вживую.** Имена событий/инструментов
   и формат вывода хуков заякорены на доках Qwen Code; до живой пробы (см.
   «Проверка под вашу сборку» в `flow-overview.md`) это допущение, а не факт.
6. **Файловые гейты — это политика на запись, не на чтение/эксфильтрацию.**
   Скопировать/прочитать enforcement-файл наружу (`cat .gigacode/… ` ) разрешено;
   защищается **редактируемость**, а не конфиденциальность.

Все шесть — **осознанные границы парсера**, а не забытые баги. Они закрываются не
`git_guard`, а слоем 5 (allow/deny + ревью).

## Как ужесточить под свой репозиторий

- **`settings.json` allow/deny.** Сузьте разрешённые инструменты/команды: запретите
  голый `bash -c`/`eval`/интерпретаторы там, где они не нужны для флоу; это
  напрямую срезает остатки 1–3.
- **Ревью человеком** на merge в защищённые ветки (gate_guard уже блокирует git
  write на protected-ветке, но процессный ревью — основной контроль качества).
- **Проба рантайма.** Выполните раздел «Проверка под вашу сборку GigaCode» в
  `flow-overview.md`, чтобы снять допущение из риска 5.
- **Не ослабляйте backstop.** `READ_ONLY_PROGS` — это allow-list; добавляя в него
  программу, убедитесь, что она действительно только читает.

Обычная сессия GigaCode превращается в управляемый процесс: агент сначала пишет спецификацию, ищет существующий код перед созданием нового, проходит проверки качества и git-безопасности — и не может завершить ход, пока работа не оформлена и не доказана.

Главная идея

Блокирует только детерминированное

Граница проведена жёстко: то, что можно проверить машиной — останавливает работу; то, что лишь похоже на проблему — только подсказывает. Никаких ложных стопов на эвристиках.

🛑

Детерминированные проверки → блок

У которых есть машинный вердикт: линтер (gate_lint), сборка (gate_build), strict-валидация спеки, полнота артефактов, git-безопасность. Зелёное или нет — третьего нет. На них флоу останавливается.

💡

Эвристики → подсказка

Где «правильность» субъективна: размер файла/функции, TODO/FIXME, Thread.sleep в тестах, «похожий символ уже есть». Это advisory: предупреждают, но никогда не стопорят работу.

⚖️

Принцип: предотвратить нарушение до того, как оно написано, дешевле, чем ловить постфактум. Поэтому правила и контекст заземляются в агента ещё до генерации кода.

Точка входа

Две команды · два режима

Запуск задачи — одна slash-команда. Или просто опишите задачу обычным запросом — скилл development-flow подхватит её сам.

✨

/develop-feature "<описание>"

Фичи и техулучшения. На входе: цель, критерии приёмки, что в скоупе / вне, ограничения.

🐛

/fix-bug "<симптом>"

Дефекты и регрессии. На входе: симптом, ожидаемое vs фактическое, шаги воспроизведения, окружение, severity.

🔍 plan-only · по умолчанию

Анализ, карта влияния и Markdown-артефакты — без правок кода. Безопасный режим по умолчанию, когда скоуп или критерии неясны.

⚙️ implement

Полный plan-only + прохождение git-safety + точечные правки + проверка поведения + PR-заметки. Включается явно или после согласования плана.

Слой контроля

Роутер + гейты

Один диспетчер router.py на все события. Маршруты — в router.config.json: новый гейт = новый файл в gates/ + строка в конфиге. Каждое решение пишется в decisions.jsonl.

Safety-critical

git_guard — самозащита флоу

Единственный safety-critical гейт. Распознаёт опасное сквозь обёртки и маскировку: chaining, env-префиксы, bash -c, кавычки, абсолютные пути, перенаправления, PowerShell-командлеты, подстановки $(...), Unicode-разделители.

  • ⛔Защищённые веткизапись на main, master, develop, release/*, hotfix/*, production, staging
  • ⛔Деструктивный gitreset --hard, clean -fd, force-push, удаление веток, remote set-url, checkout --
  • ⛔Разрушение репозиторияудаление .git / .gigacode через rm, Remove-Item, find -delete
  • ⛔Правка слоя контролялюбая запись в .gigacode/** и .git/** — агент не отключит ограничивающий его enforcement
  • ?Истина спеки · CI · секретыASKopenspec/specs, .github/workflows, secrets/, terraform/, .env
defense-in-depth · самозащита .gigacode/.git

$ rm -rf .gigacode        # → BLOCK
$ bash -c "git push -f"   # → BLOCK
$ echo x > .git/config     # → BLOCK
$ python -c "shutil...gigacode" # → BLOCK

Спека как истина + анти-дубли

OpenSpec и Search Before Create

Авторитетная спека ведётся отдельно от рабочих заметок и накапливает «текущую истину» по каждой capability. Перед новым кодом — обязательный поиск существующего.

📐 Жизненный цикл спеки

/opsx:propose
change в openspec/changes/<id>/: proposal, design, tasks, дельты требований
validate --strict
не идёт в работу, пока валидация не зелёная
/opsx:apply → archive
после мержа change архивируется в openspec/specs/

🔎 Search Before Create

перед каждым новым символом
# 1. семантический поиск (Serena)
find_symbol(name="PaymentRetry")

# fallback без Serena — ripgrep
$ rg "class PaymentRetry" -t code

# нашёл? → читай и расширяй,
#          не пиши дубликат
# для Kafka — ищи по имени топика,
# не только по имени класса

Дубликат консьюмера на тот же топик — ровно тот класс дубликатов, который болит. gate_existing_code подсказывает, если похожее уже есть.

На выходе

Артефакты задачи

Человекочитаемые заметки лежат в docs/development/<task-slug>/ и ссылаются на OpenSpec-спеку, не дублируя её.

📓

journal.md

Контекст, карта влияния, план и ход работы. Человекочитаемо, ссылается на спеку.

✅

verification.md

Какие проверки прогнали, их вывод, что пропущено и почему. Без доказательств тут — завершение блокируется.

📝

pr-summary.md

Готовый текст для PR — для ревьюера, без лишней работы.

📐

openspec/changes/<id>/

Авторитетная спека: зачем, дизайн, чеклист, дельты. После мержа → specs/.

🌿

feature-ветка

Только правки кода. Commit/push — руками или по явной команде, не автоматом.

🧾

logs/decisions.jsonl

Каждое решение каждого гейта — для тюнинга системы и разбора, почему сработало.

Усиление контекста

MCP-серверы · все с фолбэком

Каждый сервер включает дополнительную возможность. Нет сервера — graceful fallback. Для работы флоу и smoke-проверок не требуется ни один.

🧭

Serena

Семантический поиск символов через LSP — фундамент Search Before Create и индекса для инъекций.

Java надёжен (JDT LS), Kotlin-LSP слабее — проверить на своём коде.
fallback: rg по паттернам

📚

Context7

Свежие доки библиотек при написании нового кода: Spring Kafka, корутины, Reactor, JUnit 5 — против галлюцинаций API.

fallback: знания модели + ручная проверка

🕸️

Graphify

Граф знаний репо → карта модулей module-map.md, инжектится на старте. Главное — карта потоков событий через топики.

fallback: ручная карта в journal.md

Ключевой контроль

Нельзя завершить ход без оформления

При попытке завершить ход срабатывает событие Stop. Гейты смотрят на реальное рабочее дерево (git status), а не на текст сообщения — обойти, не упомянув пути, нельзя.

  • 1Спека есть и валиднаменялся production-код → существует активный OpenSpec-change, проходящий --strict
  • 2Артефакты полные и доказанныеjournal / verification / pr-summary без TODO/TBD/FIXME и с доказательствами проверок
  • 3Сборка зелёнаяесли build настроена в quality-gates.json и менялся код
🔒

Блок — постоянный

Повторные попытки завершить тоже блокируются, пока причина не устранена. Никакой автодеградации «после N попыток пропустим».

Все причины чинятся самим агентом: дописать спеку, дополнить артефакт, убрать TODO, поправить сборку. Застрявший гейт — человеком через escape hatch disableAllHooks.

Итог

Что слой контроля гарантирует

  • ✓Нельзя зашипить production-код без оформленной и валидной спеки и без полных, доказанных артефактов.
  • ✓Нельзя выполнить деструктивную git-операцию или писать на защищённой ветке без явного человеческого решения.
  • ✓Нельзя отредактировать или удалить сам enforcement-слой (.gigacode/**) или репозиторий (.git) — в т.ч. через обёртки и подстановки.
  • ✓Эвристики подсказывают, но не стопорят — дубликаты, размеры, анти-паттерны тестов остаются предупреждениями.

# Правила разработки

Разработка выполняется для одной задачи за раз.

## Инварианты процесса

- Сначала план, затем код. Изменения в `src/` без утвержденного плана в `docs/plans/` блокируются hook-ом.
- План утверждает человек: статус `Status: approved` выставляется только после явного подтверждения пользователя.
- Реализация ведется малыми шагами, каждый шаг должен оставлять код компилируемым.
- Для каждого измененного production-класса должен существовать тест. Исключение фиксируется в плане строкой `no-tests: <причина>`.
- Стиль кода и тестов задается файлами в `reference/`. Hooks автоматически подмешивают выдержки из них при редактировании Java/Kotlin-файлов.
- Детерминированные статические проверки выполняются всегда. Реальные сборка, линтеры и тесты запускаются опционально (см. `reference/build-tools.md`), их недоступность явно фиксируется как ограничение.
- Изменения файлов сборки (`build.gradle*`, `pom.xml`, wrapper) требуют отдельного подтверждения.
- Использовать `repomix`, `graphify` и Serena MCP как необязательные ускорители исследования кода. Если они недоступны, продолжать обычным поиском и фиксировать ограничение.

## Порядок агентов

1. `dev-plan` - исследование кода и план в `docs/plans/<task-slug>.md`.
2. Пользователь утверждает план.
3. `dev-implement` - реализация по шагам плана.
4. `dev-test` - тесты JUnit 5 для измененных классов.
5. `dev-review` - финальное ревью без новых правок.
6. Hooks валидируют результат на каждом этапе.

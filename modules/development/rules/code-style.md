# Code Style Rules

> **ОБРАЗЕЦ — отредактируйте под конвенции вашей команды.** Это шаблон с
> разумными значениями по умолчанию для стека Kotlin + Java. Замените примеры
> своими правилами; удалите неприменимое.
>
> Этот файл **инжектится в контекст агента** на старте сессии и при запуске
> сабагента `coder` (механизм — `gate_context_inject`, список в
> `RULE_FILES`). Держите его компактным: всё, что здесь написано, агент читает
> перед генерацией кода.

Технические идентификаторы (имена символов, пакеты, аннотации) — English/ASCII.
Пользовательские пояснения и комментарии — по language-policy проекта.

## Структура и размер

- Один публичный тип на файл; имя файла = имя типа.
- Ориентир: файл ≤ 400 строк, функция ≤ 60 строк, ≤ 3 уровней вложенности.
  Жёсткие лимиты — в `.gigacode/quality-gates.json` (`clean_code`); это
  advisory-ориентир, не отговорка плодить «god-классы».
- Публичный API модуля держать узким; детали — `internal`/`private`.

## Именование

- Классы/интерфейсы — `PascalCase`; функции/свойства — `camelCase`;
  константы — `UPPER_SNAKE_CASE`.
- Имя отражает намерение, а не тип (`retryPayment`, не `doProcessing2`).
- Без венгерской нотации и аббревиатур, кроме общепринятых (`id`, `url`, `http`).

## Kotlin: null-safety и иммутабельность

- По умолчанию `val`, не `var`; неизменяемые коллекции (`List`, не
  `MutableList`) в сигнатурах.
- Не использовать `!!`; для отсутствующего значения — `?:`, ранний возврат
  или явная ошибка домена.
- Предпочитать `data class` для DTO, `sealed`-иерархии для замкнутых состояний.
- Не «глотать» `null`/исключения молча — логировать с контекстом или пробрасывать.

## Ошибки и логирование

- Кидать доменные исключения, а не generic `RuntimeException`.
- Не ловить `Throwable`/`Exception` «на всякий случай»; ловить узко.
- Логи структурные, с корреляционными полями; без секретов/PII в сообщениях.

## Асинхронность и корутины

- У внешних вызовов — явный таймаут (`withTimeout`) и стратегия отмены.
- Не блокировать корутины (`Thread.sleep`, блокирующий IO без `Dispatchers.IO`).
- Прокидывать `CoroutineContext`/cancellation, не запускать «осиротевшие» job.

## Kafka

- Перед новым consumer/producer — проверить существующие по **имени топика**,
  не только по имени класса (см. `existing_code` гейт): дубликат consumer на
  тот же топик — частый источник багов.
- Имена топиков и ключи сериализации — в конфиге/константах, не «магией» по коду.
- Идемпотентность обработчиков и явная политика на ошибку обработки (retry/DLQ).

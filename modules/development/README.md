# Модуль разработки GigaCode

Модуль содержит проектную конфигурацию GigaCode для разработчиков Java/Kotlin: плановая разработка с человеком в петле, детерминированные проверки качества плана и кода через hooks и компетенции по JUnit 5.

Рабочий контур использует проектную конфигурацию GigaCode и каталог `.gigacode/`.

## Состав модуля

- `.gigacode/settings.json` - настройки проекта, разрешения и подключение hooks на четыре события.
- `.gigacode/hooks/router.py` - единый роутер hooks: принимает событие, подбирает проверки по `router_config.json` и объединяет решения.
- `.gigacode/hooks/checks/` - модули проверок: preflight промпта, гейт плана, защита от опасных команд, инъекция контекста, статический анализ кода, финальная валидация.
- `.gigacode/hooks/hooklib/` - общая библиотека: парсинг событий, автодетект Gradle/Maven, классификация исходников, валидация плана.
- `.gigacode/agents/` - четыре субагента: `dev-plan`, `dev-implement`, `dev-test`, `dev-review`.
- `.gigacode/skills/` - навыки `feature-development` и `junit-testing`.
- `.gigacode/commands/` - slash-команды `/develop` и `/write-tests`.
- `reference/` - правила по JUnit 5, стилю Kotlin/Java, мокам, assert-ам и командам сборки. Hooks автоматически подмешивают выдержки (`## Digest`) при редактировании соответствующих файлов.
- `rules/` - правила процесса, формата плана, тестирования и именования веток.
- `docs/templates/plan-template.md` - шаблон плана задачи.
- `docs/serena-mcp.sample.json` - образец подключения Serena MCP.

## Требования

- GigaCode CLI, доступный как команда `gigacode`.
- Git.
- Python 3, доступный как команда `python`.
- Опционально: Gradle или Maven (предпочтительно wrapper в репозитории) для реальной сборки и тестов.
- Опционально: `ktlint`, `detekt`, `checkstyle`, `spotless` - используются, только если подключены в файле сборки проекта.
- Опционально: `repomix` для компактной карты репозитория.
- Опционально: `graphify` для минимального подграфа зависимостей изменяемой фичи.
- Опционально: Serena MCP для семантической навигации по коду (см. ниже).

Если опциональные инструменты не установлены, GigaCode продолжает работу обычным поиском и явно фиксирует ограничение.

## Быстрый старт на Windows

```powershell
git clone <repo-url>
cd <repo>\modules\development
.\scripts\smoke-check.ps1
gigacode
```

Затем выполните внутри GigaCode:

```text
/develop DEV-123 добавить лимит на блокировку карты
```

## Быстрый старт на Linux

```bash
git clone <repo-url>
cd <repo>/modules/development
bash scripts/smoke-check.sh
gigacode
```

## Рабочий процесс

1. Hook `prompt_preflight` проверяет полноту запроса (описание задачи, ссылка на Jira или явное `без jira`).
2. Hook `prompt_context` подмешивает сводку по системе сборки и указатели на справочные файлы.
3. `dev-plan` исследует код (Serena/graphify/repomix при наличии) и пишет план в `docs/plans/<task-slug>.md` со статусом `draft`.
4. Пользователь утверждает план - статус меняется на `approved`. Без этого hook `plan_gate` останавливает правки `src/`.
5. `dev-implement` реализует план малыми шагами. Hook `posttool_static` проверяет каждый записанный файл.
6. `dev-test` пишет JUnit 5 тесты для измененных классов.
7. `dev-review` делает финальное ревью: соответствие диффа плану, стиль, тесты.
8. На Stop hook `stop_final` повторяет статические проверки по всем измененным файлам и проверяет наличие тестов; `stop_build` опционально запускает реальную сборку.

## Только тесты

```text
/write-tests src/main/kotlin/com/example/CardService.kt
```

## Реальная сборка в Stop-hook

По умолчанию Stop-hook выполняет только статические проверки. Запуск компиляции, тестов и линтеров включается одним из способов:

- переменная окружения `GIGACODE_DEV_RUN_BUILD=1`;
- строка `build-check: on` в утвержденном плане.

При недоступности инструментов или таймауте hook пропускает запуск и фиксирует ограничение, не блокируя работу.

## Подключение Serena MCP

Образец конфигурации лежит в `docs/serena-mcp.sample.json`. Перенесите блок `mcpServers` в `.gigacode/settings.json`, если на машине установлен `uv` и языковой сервер для Java/Kotlin. Агенты используют семантические инструменты Serena при наличии и автоматически переходят на обычный поиск без них.

## Поддержка событий hooks

Модуль подключает четыре события: `UserPromptSubmit`, `PreToolUse`, `PostToolUse`, `Stop`. События `PreToolUse`/`PostToolUse` наследуются GigaCode от Qwen Code; если ваша сборка их не поддерживает, удалите соответствующие блоки из `settings.json` - все блокирующие гарантии (гейт плана, статические проверки, наличие тестов) продублированы на этапе Stop, роутер и проверки изменений не требуют.

## Ограничение размера субагентов

Каждый файл субагента должен оставаться короче 10 000 символов. Повторно используемые детали выносятся в `reference/` и `rules/`.

## Адаптация для командного репозитория

Используйте этот модуль как корень проекта для разработчиков: скопируйте `.gigacode/`, `reference/`, `rules/` и `docs/templates/` в корень вашего Java/Kotlin-репозитория. Hooks рассчитаны на стандартную структуру `src/main/...` и `src/test/...`; пути проверок настраиваются в `.gigacode/hooks/router_config.json`.

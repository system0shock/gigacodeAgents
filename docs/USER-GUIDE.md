# GigaCode Developer Flow — руководство пользователя

Этот шаблон превращает обычную сессию GigaCode в управляемый процесс разработки:
агент сначала пишет спецификацию, ищет существующий код перед тем как писать
новый, проходит проверки качества и git-безопасности, и **не может завершить ход**,
пока работа не оформлена и не проверена. Эвристики только подсказывают;
блокируют лишь детерминированные проверки (git, валидатор спеки, линтер, сборка).

> Язык по умолчанию — русский (вопросы, блокеры, артефакты). Технические
> идентификаторы — пути, команды, имена веток/хуков/символов — не переводятся.

Содержание:

## Быстрый старт

### Шаг 1. Установите зависимости

**Обязательные** (без них флоу не работает):

| Зависимость | Зачем | Установка / проверка |
|---|---|---|
| **GigaCode CLI** | хост, который запускает хуки и команды | по инструкции вашей сборки GigaCode |
| **Git** | гейты читают рабочее дерево; git_guard защищает репозиторий | `git --version` (должен быть в PATH) |
| **Python 3** | роутер и все гейты — это Python | `python --version` (должен вызываться как `python`) |
| **PowerShell** (Windows) или **Bash** (Linux/macOS) | только для smoke-проверок | предустановлены |

**Рекомендуемая** (флоу работает и без неё, но теряет strict-проверку спеки):

```bash
npm install -g @fission-ai/openspec@1.4.1
```

Это **OpenSpec CLI** — валидирует спецификации (`openspec validate --strict`).
Без него гейт `gate_spec_structure` не падает, а пропускает валидацию с записью
в журнал (вы увидите пометку «strict-валидация пропущена»).

**Опциональные** (каждая включает дополнительную возможность; без неё — graceful
fallback):

<details>
<summary><b>Serena</b> — семантический поиск кода (поиск символов через LSP)</summary>

```bash
pip install uv                      # менеджер окружений; или см. https://docs.astral.sh/uv/
uv tool install -p 3.13 serena-agent
serena init                         # инициализирует LSP-бэкенд
serena project create               # создаёт .serena/project.yml в текущем репо
```
Без Serena агенты ищут существующий код через `rg` (текстовый поиск).
Замечание: Java покрыта надёжно (JDT LS), Kotlin-LSP слабее — проверьте на своём коде.
</details>

<details>
<summary><b>Context7</b> — свежая документация библиотек (Spring Kafka, JUnit 5, корутины)</summary>

Нужен **Node.js 18+**. Сервер поднимается автоматически из `settings.json` через
`npx -y @upstash/context7-mcp`. Опционально — API-ключ (выше rate limit):
добавьте `--api-key <ключ>` в `args` (ключи в репозиторий не коммитим).
Без Context7 агент полагается на знания модели + ручную проверку API.
</details>

<details>
<summary><b>Graphify</b> — карта модулей репозитория (инжектится в контекст)</summary>

```bash
pip install graphifyy
/graphify .                              # построить граф репозитория (повтор: /graphify . --update)
python scripts/build_module_map.py       # граф -> .gigacode/context/module-map.md
```
Без Graphify файла карты просто нет — гейт молча пропускает секцию (можно вести
карту вручную в том же файле).
</details>

### Шаг 2. Поставьте шаблон в репозиторий

Шаблон — это набор каталогов, которые копируются в корень вашего проекта:

```text
.gigacode/   настройки, хуки (роутер + гейты), скиллы, команды, агенты
openspec/    конфиг и каталог спецификаций
rules/       правила: git-safety, language-policy, development-flow, openspec
scripts/     билдер карты модулей + smoke-проверки
```

После копирования адаптируйте под команду (детали — в `README.md`):
- `rules/git-safety.md` — ваши защищённые ветки и пути;
- `.serena/project.yml` — `project_name` и `ignored_paths`;
- `.gigacode/quality-gates.json` — команды линта/сборки/тестов (см. [ниже](#настройка-под-ваш-стек)).

### Шаг 3. Проверьте установку

```powershell
.\scripts\smoke-check.ps1      # Windows
```
```bash
bash scripts/smoke-check.sh    # Linux / macOS
```
Smoke-проверки оффлайн: не требуют GigaCode, MCP-серверов, сети и доступов.
Должно завершиться строкой `Smoke check passed`.

> Опционально, но рекомендуется один раз: сверьте имена событий/тулов вашей
> сборки GigaCode с матчерами в `router.config.json` — см. раздел «Проверка
> семантики хуков» в `README.md` (`hook_probe.py`).

### Шаг 4. Запустите первую задачу

Откройте сессию GigaCode в корне репозитория (`gigacode`) и наберите:

```text
/develop-feature plan-only payment retry; критерии приёмки: повторять упавший вызов провайдера один раз после транзиентного таймаута
```

Режим `plan-only` (по умолчанию) подготовит спецификацию и план **без правок
кода**. Когда план устроит — повторите в режиме `implement` либо подтвердите
переход. Для багов — `/fix-bug`:

```text
/fix-bug plan-only карта блокируется по таймауту; ожидание: пользователь видит финальный статус; факт: запрос висит после таймаута провайдера
```

## Возможности

### Команды

| Команда | Для чего | Режимы |
|---|---|---|
| `/develop-feature "<описание>"` | фичи и техулучшения | `plan-only` (по умолчанию) · `implement` |
| `/fix-bug "<симптом>"` | дефекты и регрессии | `plan-only` · `implement` |

Обе используют скилл `development-flow`. Можно и просто описать задачу обычным
запросом — скилл подхватит её сам. `plan-only` — анализ и план без изменения
кода; `implement` — план, git-safety проверки, точечные правки, проверка
поведения и подготовка PR-заметок.

### Слой контроля: роутер + гейты

Все хуки идут через единый диспетчер `.gigacode/hooks/router.py`. Маршруты
описаны в `router.config.json` (никакого редактирования роутера — новый гейт это
просто новый файл в `gates/` + строка в конфиге). Каждое решение пишется в
`.gigacode/logs/decisions.jsonl`.

| Гейт | Когда | Что делает |
|---|---|---|
| `git_guard` | перед Bash/Shell и записью файла | **блок** деструктивного git и разрушения/правок репозитория |
| `gate_spec_structure` | запись файла, завершение | **блок**: запись в `openspec/specs`, невалидный change, код без спеки |
| `validate_development_output` | завершение | **блок**: неполные/«пустые» артефакты текущей задачи |
| `gate_build` | завершение | **блок** при провале сборки (если настроена и менялся код) |
| `gate_lint` | после записи файла | **блок** при ненулевом exit линтера (если настроен) |
| `gate_clean_code` | после записи файла | **advisory**: размер файла/функции, TODO/FIXME, `Thread.sleep` в тестах |
| `gate_existing_code` | перед созданием нового файла | **advisory**: «похожий символ/Kafka-топик уже есть в X» |
| `gate_context_inject` | старт сессии / субагента, запрос | инъекция правил, карты модулей, активных changes |
| `preflight_check` | отправка запроса | **блок** пустой/невалидной dev-задачи |

Принцип: **блокирует только детерминированное** (git, валидатор спеки, линтер,
сборка). Эвристики (`clean_code`, `existing_code`) — только предупреждение,
никогда не блок.

### git-безопасность

`git_guard` — единственный safety-critical гейт. Он блокирует:
- запись на **защищённых ветках** (`main`, `master`, `develop`, `release/*`,
  `hotfix/*`, `production`, `staging`, `uat`);
- **деструктивный git**: `reset --hard`, `clean -fd`, force-push (`-f`, `+ref`,
  `--mirror`), удаление веток, `remote set-url`, `reflog expire/delete`,
  `gc --prune`, `checkout --`, `worktree remove`, `stash clear` и др.;
- **разрушение репозитория**: удаление `.git` или `.gigacode` (через `rm`,
  `Remove-Item`, `find -delete`, `rmdir`);
- **правку самого слоя контроля и истины спеки**: запись в `.gigacode/**`,
  `.git/**`, `openspec/specs|archive` — чтобы агент не мог отредактировать или
  отключить enforcement, который его ограничивает.

Распознавание работает **сквозь обёртки и маскировку**: chaining (`;`, `&&`,
`|`), env-префиксы, `sudo`/`timeout`/`nice`, `bash -c`/`cmd /c`, `command`/`eval`,
кавычки, абсолютные пути, `.exe`, перенаправления (`>`, `>>`, `1>`, `>|`),
PowerShell-командлеты (`Set-Content`, `Out-File`), подстановки (`$(...)`,
backticks) и Unicode-разделители (NBSP). На защищённые пути CI/секретов/инфры
(`.github/workflows`, `secrets/`, `terraform/`, `.env`) гейт отвечает **ask**
(требует подтверждения), а не блок.

> v1 не делает auto-commit и auto-push. «PR-ready» = подготовлены заметки для
> ревьюера и доказательства проверки; коммит/пуш — руками или по явной команде.

### Спецификации (OpenSpec)

Авторитетная спека ведётся через OpenSpec и живёт отдельно от рабочих заметок:

- изменение оформляется как **change** в `openspec/changes/<id>/` (`proposal.md`,
  `tasks.md`, дельты требований);
- не идёт в работу, пока `openspec validate <id> --strict` не зелёный;
- после завершения **архивируется** (`openspec archive`) в `openspec/specs/` —
  так накапливается «текущая истина» по каждой capability.

Рабочий процесс скиллов: `/opsx:propose "<идея>"` → заполнить и
`openspec validate <id> --strict` → `/opsx:apply` → `/opsx:archive`. Правила
формата — в `rules/openspec.md`.

### Артефакты разработки

Человекочитаемые заметки задачи лежат в `docs/development/<task-slug>/` и
**ссылаются** на OpenSpec-спеку, не дублируя её:

| Файл | Что внутри |
|---|---|
| `journal.md` | контекст, карта влияния, план и ход работы |
| `verification.md` | какие проверки прогнали, результаты, что пропущено и почему |
| `pr-summary.md` | готовый текст для PR |

### MCP-серверы и карта модулей

- **Serena** — семантический поиск символов (фундамент «ищи перед тем как
  создавать»); fallback — `rg`.
- **Context7** — свежие доки библиотек против галлюцинаций API; fallback —
  знания модели + ручная проверка.
- **Graphify → `module-map.md`** — компактная карта модулей (и, что важно для
  Kafka-стека, событийных потоков producer → topic → consumer), инжектится на
  старте сессии; fallback — ручная карта.

Все три опциональны: флоу и smoke-проверки работают без них.

## Как устроен флоу

Пошагово — что происходит в сессии и где включается контроль.

**1. Старт сессии.** Хук `SessionStart` инжектит в контекст правила проекта
(git-safety, language-policy, development-flow) и карту модулей. Агент сразу
знает ограничения и архитектуру.

**2. Запуск задачи.** Вы вводите `/develop-feature` / `/fix-bug` (или обычный
запрос). Хук `preflight_check` проверяет, что в запросе есть минимум вводных:
для фичи — цель и критерии приёмки; для бага — симптом, ожидаемое vs фактическое,
шаги. Если их нет — **блок с просьбой уточнить** (агент не фантазирует). Вопросы
задаёт только главная сессия.

**3. План (plan-only).** Готовится OpenSpec-change в `openspec/changes/<id>/`:
зачем, дизайн, чеклист задач, дельты требований. Спека проходит
`openspec validate --strict`. На этом этапе кода ещё нет.

**4. Реализация (implement).** Перед каждым новым классом/функцией — **поиск
существующего** (Serena или `rg`); `gate_existing_code` подсказывает, если
похожий символ или Kafka-топик уже есть. Во время правок:
- `git_guard` (PreToolUse) блокирует опасные команды и запись в защищённые пути;
- `gate_spec_structure` не даёт писать напрямую в `openspec/specs`;
- после каждой записи (PostToolUse) — `gate_lint` (блок при ошибке линтера),
  `gate_clean_code` (предупреждения), strict-валидация затронутого change.

**5. Проверка.** Прогоняются проверки, доказательства фиксируются в
`verification.md` (команды и их вывод). Заявление «всё прошло» без доказательств
в `verification.md` будет заблокировано на завершении.

**6. Завершение (ключевой контроль).** При попытке завершить ход срабатывает
событие `Stop`. Гейты смотрят на **реальное рабочее дерево** (`git status`), а
**не на текст сообщения** — нельзя «обойти» проверку, просто не упомянув пути.
Чтобы завершить, должны выполняться три условия:

1. если менялся production-код — существует активный OpenSpec-change, и он
   проходит strict-валидацию (`gate_spec_structure`);
2. для текущего изменения есть полные артефакты `docs/development/<slug>/`
   (`journal.md` / `verification.md` / `pr-summary.md`), без placeholder-маркеров
   (`TODO`/`TBD`/`FIXME`) и с доказательствами проверок
   (`validate_development_output`);
3. сборка зелёная, если она настроена в `quality-gates.json` (`gate_build`).

Если условие не выполнено — **блок**, и он **постоянный**: повторные попытки
завершить ход тоже блокируются, пока причина не устранена (никакой
автодеградации «после N попыток пропустим»). Все эти причины **чинятся самим
агентом** — дописать спеку, дополнить артефакты, поправить сборку.

**7. Архивирование.** После мержа `openspec archive` переносит change в
`openspec/specs/` — «текущая истина» по capability обновляется.

### Что слой контроля гарантирует

- нельзя зашипить production-код без оформленной и валидной спеки и без полных,
  доказанных артефактов;
- нельзя выполнить деструктивную git-операцию или писать на защищённой ветке без
  явного человеческого решения;
- нельзя отредактировать или удалить сам enforcement-слой (`.gigacode/**`),
  истину спеки (`openspec/specs`) или репозиторий (`.git`) — в т.ч. через
  обёртки, перенаправления и подстановки;
- эвристики (дубликаты, размеры, анти-паттерны тестов) **подсказывают**, но не
  стопорят работу.

## Настройка под ваш стек

Языковая специфика живёт в одном файле `.gigacode/quality-gates.json`, не в коде
гейтов. Пример для Kotlin/Java + Gradle:

```json
{
  "lint": [
    { "command": "./gradlew ktlintCheck detekt", "applies_to": ["**/*.kt", "**/*.kts"], "timeout_seconds": 300 },
    { "command": "./gradlew checkstyleMain", "applies_to": ["**/*.java"], "timeout_seconds": 300 }
  ],
  "build": { "command": "./gradlew compileKotlin compileTestKotlin", "timeout_seconds": 600 },
  "test":  { "command": "./gradlew test" },
  "clean_code": { "max_file_lines": 400, "max_function_lines": 60, "placeholder_markers": ["TODO", "FIXME", "XXX"] }
}
```

- Пустая `command` = гейт пропускается без записи (silent allow) — пока команды
  не настроены, работа в репозитории не блокируется.
- На Windows используйте `gradlew.bat ...`, на Linux/macOS — `./gradlew ...`.
- `test.command` исполняет агент `verifier` (сами гейты тесты не гоняют).

Полная справка по конфигурации и адаптации — в `README.md`.

## Когда что-то блокирует

1. **Прочитайте причину.** Гейт всегда возвращает текст: чего не хватает и что
   сделать (например, «изменён production-код, но нет активного
   `openspec/changes/<id>/`» или «Placeholder marker найден в …»).
2. **Устраните и повторите.** Причины детерминированы и чинятся: дописать спеку,
   заполнить артефакт, убрать `TODO`, поправить сборку. Следующее завершение
   пройдёт — счётчик блоков сбрасывается на первом успешном `Stop`.
3. **Деструктивный git заблокирован специально.** Если действительно нужен,
   например, `reset --hard` — это ручная человеческая операция вне агента.
4. **Хуки сломались?** Временный аварийный выключатель — `"disableAllHooks": true`
   в `.gigacode/settings.json` (правит человек: для агента этот файл защищён).
   Это последнее средство — после почините причину и верните хуки.

Журнал всех решений — `.gigacode/logs/decisions.jsonl`; по нему удобно тюнить
систему и понимать, почему гейт сработал.

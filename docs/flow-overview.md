# GigaCode Developer Flow: полное описание

Обзор флоу разработки с шаблоном GigaCode Developer Template: запуск, входные
требования, артефакты, хуки, MCP-серверы и покрытие стека (Kotlin, Java,
JUnit 5, Kafka, асинхрон).

## 1. Запуск

Шаблон ставится копированием в репозиторий: `.gigacode/` (настройки, агенты,
скиллы, хуки), `openspec/`, `rules/`, `scripts/`. Никакой установки кроме
Python для хуков и (опционально) MCP-серверов.

Дальше всё происходит в обычной сессии GigaCode:

1. **Старт сессии** → хук `SessionStart` инжектит в контекст правила проекта
   (git-safety, language policy, development-flow) и индекс модулей/символов.
2. Разработчик запускает работу командой:
   - `/develop-feature "<описание>"` — фичи и техулучшения;
   - `/fix-bug "<симптом>"` — дефекты и регрессии;
   - или обычным запросом — скилл `development-flow` подхватывает его сам.
3. Хук `UserPromptSubmit` (preflight) проверяет запрос: если это dev-задача
   без минимальных вводных — блокирует с просьбой уточнить, а не даёт агенту
   фантазировать.

Режим по умолчанию — `plan-only` (анализ и план без правок кода). `implement`
— явно или после согласования плана.

## 2. Что требуется на входе

**Для фичи:** цель, критерии приёмки, что в скоупе / что вне, ограничения.

**Для бага:** симптом, ожидаемое vs фактическое поведение, шаги
воспроизведения, окружение, severity.

Чего не хватает — главная сессия спрашивает до начала работы (intake не
делегируется субагенту: только главная сессия может задавать вопросы
пользователю); preflight не пропустит пустой запрос дальше.

**Опциональный контекст** (есть — используется, нет — фиксируется как
ограничение): выход Graphify (граф репозитория), проектная аналитика,
поднятые MCP-серверы.

## 3. Конвейер агентов

```
intake (главная сессия) → repo-context → OpenSpec propose
   → [STOP в plan-only] → coder → verifier
```

- Intake и план живут в главной сессии: вопросы пользователю может задавать
  только она, а план — это артефакты OpenSpec-change, не отдельный документ.
- `repo-context` собирает project intelligence и строит карту влияния.
  **Для нашего стека критично:** кроме графа вызовов он обязан построить карту
  событийных потоков — producer → topic → consumer'ы, — потому что Kafka
  разрывает статический call-graph, и «кто читает этот топик» из кода по
  символам не видно.
- Спека оформляется как OpenSpec-change (`proposal.md`, `design.md`,
  `tasks.md`, дельты требований) и не идёт в работу, пока
  `openspec validate --strict` не зелёный.
- `coder` получает в инъекции найденные существующие символы по зоне задачи и
  обязан выполнить Search Before Create перед каждым новым
  классом/функцией/модулем.
- `verifier` гоняет проверки, фиксирует доказательства и готовит PR-сводку.

## 4. Артефакты на выходе

| Артефакт | Где | Что это |
|---|---|---|
| OpenSpec-change | `openspec/changes/<id>/` | Авторитетная спека: зачем, дизайн, чеклист задач, дельты требований. После мержа архивируется в `openspec/specs/` — накапливается «текущая истина» по capability |
| Журнал задачи | `docs/development/<task>/journal.md` | Контекст, план, ход работы — человекочитаемо, ссылается на спеку |
| Верификация | `docs/development/<task>/verification.md` | Что прогнали, результаты, что пропущено и почему |
| PR-сводка | `docs/development/<task>/pr-summary.md` | Готовый текст для PR |
| Код | feature-ветка | Только правки; commit/push — руками или по явной команде |
| Журнал гейтов | `.gigacode/hooks/decisions.jsonl` | Каждое решение каждого гейта — для тюнинга системы |

## 5. Как работают хуки

Один диспетчер `router.py` зарегистрирован на все события. GigaCode передаёт
ему JSON события на stdin (имя события, тул, параметры, cwd); роутер по
конфиг-таблице `router.config.json` находит подходящие гейты, прогоняет их и
возвращает агрегированное решение: deny/block, инъекция контекста или пропуск.

| Событие | Гейт | Действие |
|---|---|---|
| `SessionStart`, `SubagentStart(coder)` | context_inject | инъекция правил + индекса символов |
| `UserPromptSubmit` | preflight | блок невалидного запроса |
| `PreToolUse(Bash)` | git_guard | блок: защищённые ветки, `reset --hard`, force-push и т.п. |
| `PreToolUse(Write/Edit → openspec/)` | spec_structure | блок невалидной спеки |
| `PreToolUse(Write/Edit → код)` | existing_code | **advisory**: «похожий символ уже есть в X» |
| `PostToolUse(Write/Edit)` | lint, clean_code | блок по линтеру изменённых файлов; эвристики — предупреждение |
| `Stop` | build, validate_output | блок до зелёного билда и полных артефактов, максимум 2 ретрая |

Предохранители: лимит ретраев против бесконечных циклов на Stop,
latency-бюджет ≤ 200 мс на pre-проверку, при падении гейта — внятная причина и
escape hatch, журналирование всех решений.

Принцип: **блокируем детерминированное (git, валидатор, билд), эвристики
только предупреждают.**

## 6. MCP-серверы

| Сервер | Зачем | Фолбэк |
|---|---|---|
| **Serena** | Семантический поиск символов через LSP — фундамент Search Before Create и индекса для инъекций. Java покрыт зрелым Eclipse JDT LS; **Kotlin LSP заметно слабее — обязательная проверка на нашем коде до ставки на него** | `rg` по паттернам символов |
| **Context7** | Свежие доки библиотек при написании нового кода: Spring Kafka, kotlinx.coroutines, Reactor, JUnit 5 — против галлюцинаций API | знания модели + ручная проверка |
| **Graphify** | Граф знаний репозитория; главное для нас — карта событийных потоков через топики, которой нет в символьном поиске | ручная карта в journal.md |

Все серверы опциональны: ничего не требуется для работы флоу и
smoke-проверок.

## 7. Покрытие стека: Kotlin, Java, JUnit 5, Kafka, асинхрон

Языковая специфика живёт в одном файле `.gigacode/quality-gates.json`, не в
коде гейтов:

```json
{
  "lint":  { "command": "./gradlew ktlintCheck detekt", "applies_to": ["**/*.kt"] },
  "lint_java": { "command": "./gradlew checkstyleMain", "applies_to": ["**/*.java"] },
  "build": { "command": "./gradlew compileKotlin compileTestKotlin" },
  "test":  { "command": "./gradlew test --tests <scope>" },
  "clean_code": { "max_file_lines": 400, "max_function_lines": 60 }
}
```

Специфика Kafka и асинхрона закрывается в трёх местах:

1. **repo-context**: обязательная карта producer → topic → consumer (поиск по
   `@KafkaListener`, `KafkaTemplate`, конфигам топиков) — иначе оценка влияния
   изменения консьюмера всегда будет неполной.
2. **existing_code**: перед созданием нового консьюмера/хендлера — поиск
   существующих по имени топика, не только по имени класса. Дубликат
   консьюмера на тот же топик — ровно тот класс дубликатов, который болит.
3. **verifier + clean_code**: анти-паттерны асинхронных тестов —
   `Thread.sleep` в тестах (предупреждение, требовать Awaitility), отсутствие
   таймаутов у корутин/фьюч, интеграционные Kafka-тесты без Testcontainers.
   Это эвристики уровня advisory, конкретный список правил — настройка в
   `quality-gates.json`.

**Честные ограничения:** Kotlin-поддержка Serena требует проверки на реальном
коде (Java — надёжно); гейты не докажут корректность асинхронной логики — они
ловят анти-паттерны и принуждают к процессу, а конкурентные баги по-прежнему
ловятся тестами и ревью.

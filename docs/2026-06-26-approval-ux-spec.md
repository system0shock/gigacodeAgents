# Approval UX — три поверхности подтверждения (отдельная задача)

> **Назначение:** спецификация UX человеческих чекпоинтов флоу. Вынесено в
> отдельную задачу (нужны существенные правки). По этому документу её можно
> начать без повторного проектирования.
>
> **Целевое место:** `docs/2026-06-26-approval-ux-spec.md`
> **Контекст:** заменяет/надстраивает текущий v1 — ручной
> `python .gigacode/hooks/confirm.py <stage> <slug>` во втором терминале.
> **Предусловие, делающее это возможным:** в боевом контуре **yolo выключен** →
> нативный `ask` нельзя авто-одобрить, нужен живой человек.
> **Статус:** design / к реализации (отдельная задача).
> **Язык:** ru-проза; идентификаторы — English/ASCII.

---

## 1. Инвариант (не меняется ни при каком UX)

**Durable-маркер `.gigacode/approvals/<slug>/<stage>.ok` — единственный источник
истины** подтверждения. Его читают `gate_stage_order` (PreToolUse + Stop-инвариант)
и observer. Он даёт аудит и «не пере-спрашивать». **Агент создать его не может**
(out-of-band). Любая новая поверхность — лишь ещё один **агенто-неподделываемый
способ создать тот же маркер**; gate всегда сверяет маркер независимо.

Угрозы, которые UX обязан сохранять закрытыми:
- агент не одобряет сам (yolo off → `ask` требует человека);
- агент не запускает CLI (git_guard режет команду, называющую `.gigacode`-путь);
- агент не имитирует `POST /approve` как человека (действие на стороне observer,
  не агентский tool-call);
- shell-обход (запись артефакта мимо file-tool) ловит канал-независимая
  **Stop-инвариант** `gate_stage_order` (уже реализована).

## 2. Поверхность A — inline `ask`

Чекпоинт-предикаты переключаются `block → ask` (флаг в stages.json, напр.
`checkpoint_mode: ask`, дефолт ask в боевом контуре). Поскольку yolo off, харнес
спрашивает живого человека прямо в потоке; `reason` несёт understanding/scope-
рестейт («прочитай и подтверди понимание задачи/границ», а не просто «разрешить
запись»).

Чтобы inline-одобрение было **durable** (а не one-time, иначе пере-спрашивает и
не виден Stop-инварианту): **PostToolUse-компаньон `gate_approval_record`** — когда
одобренная человеком запись чекпоинт-артефакта реально легла, он пишет маркер.
Итог: «одобрил inline → стадия подтверждена насовсем».

⚠️ Покрывает только file-tool канал. Shell-запись `ask` не словит → бэкстоп —
Stop-инвариант (block на Stop при несоответствии). Inline-путь = удобство, не
замена durable-маркеру.

## 3. Поверхность B — CLI-команда

Дружелюбная обёртка вместо голого питона: напр. `gigacode approve <stage> <slug>`
(`bin/`-скрипт или подкоманда qwen) → пишет тот же маркер через логику `confirm.py`.
Out-of-band, git_guard-защищена (агент не запустит). Для случаев без дашборда /
скриптовых сценариев / CI-ассистента-человека.

## 4. Поверхность C — кнопка в observer

Рядом с активным блокером дашборд показывает «Approve» → `POST /approve {stage,
slug}` → сервер observer пишет маркер. Observer остаётся **read-model + одно
gated-действие через штамп-spine** (app предлагает → пишет маркер → gate независимо
валидирует — «control-plane через stamp», ADR-6 contested). Bind остаётся строго
`127.0.0.1`; ноль phone-home. Это ЕДИНСТВЕННОЕ отступление observer от чистого
read-only — намеренное и через тот же маркер.

## 5. План реализации (для отдельной задачи)

- `gate_stage_order`: чекпоинт `block → ask` под config-флагом; дефолт ask.
- Новый `gate_approval_record` (PostToolUse на intake/contract-артефакты): успешная
  запись → durable-маркер. Тест на агенто-неподделываемость (shell-запись маркер
  НЕ создаёт; Stop-инвариант её ловит).
- CLI: `bin/gigacode-approve` (или qwen-подкоманда) → маркер.
- observer: `POST /approve` (валидирует stage/slug, пишет маркер, fail-closed на
  кривой ввод) + кнопка в `observer.html` у `blocker.command`.
- Тесты: ask-режим разблокирует после человека; PostToolUse пишет маркер; shell
  всё ещё block на Stop; `POST /approve` пишет маркер и отвергает мусор; агент не
  может ничего из трёх.

## 6. Открытые вопросы для задачи

- Текст `ask`-промпта: как уместить understanding-рестейт в одну строку approval'а?
- Где живёт CLI-бин (в шаблоне `.gigacode/` агент его не запустит — git_guard; значит
  вне `.gigacode`, напр. `bin/` проекта или глобальная qwen-подкоманда).
- `POST /approve` аутентификация: достаточно ли loopback-bind, или нужен токен
  (на shared-машине/Citrix loopback виден другим локальным юзерам — модель угроз).
- Откат подтверждения (un-approve / amend) — отдельная кнопка/команда, удаляющая
  маркер (= повторный review).

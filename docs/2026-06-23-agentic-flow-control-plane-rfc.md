# RFC: Agentic Flow Control Plane — контракты, петли, наблюдаемость

> **Назначение:** рабочий документ, по которому ведём дальнейшую разработку
> агентного флоу GigaCode. Консолидирует идеи и фиксы из проектного обсуждения.
> Не план задача-за-задачей — это design-RFC: решения + обоснования + бэклог.
>
> **Целевое место в репо:** `docs/superpowers/specs/2026-06-23-agentic-flow-control-plane-rfc.md`
> **Статус:** draft / к ревью построчно
> **Язык:** ru-проза; технические идентификаторы (пути, поля, события, имена
> гейтов) — English/ASCII, не переводятся.

---

## 1. Контекст и текущее состояние

Репозиторий — система GigaCode-шаблонов на общей архитектуре hook-router + гейты.
На `master` слиты два модуля:

- `modules/analytics` — reverse-analysis (зрелый).
- `modules/development` — feature/bug-флоу (coder / repo-context / verifier,
  OpenSpec-lifecycle скиллы, Node-лончер `run-hook.cjs`).

Ядро enforcement: единый `router.py` диспетчит гейты из `gates/` по
`router.config.json`, агрегирует решения (`block > ask > allow`), журналит в
`decisions.jsonl`, деградирует повторные Stop-блоки после `stop_block_budget`.
`git_guard.py` — закалённый security-гейт (защита от обхода через wrappers,
aliases, redirects, unicode-сепараторы).

Этот RFC надстраивает над этим спином три слоя: **объявленный scope (контракт)**,
**детерминированные петли**, **наблюдаемость без остановки**. И фиксирует
технический долг, который иначе подорвёт эти надстройки.

## 2. Принципы проектирования (инварианты)

Любое решение ниже подчинено им. При конфликте — принцип побеждает.

- **P1. Спин = декларация + детерминированный гейт + журнал.** Намерение
  становится объявленным машиночитаемым артефактом; гейт проверяет;
  журнал фиксирует. (Контракт, план, вердикт — все по этому шаблону.)
- **P2. Управляемость над магией.** Безлимитного «крути, пока не похоже на
  готовое» нет. Любая петля: счётчик в state + журналируемый вердикт +
  принудительный выход к человеку на бюджете.
- **P3. Контроль без ожидания.** Прозрачность даётся наблюдаемостью и
  прерываемостью, не новыми барьерами. Барьеры — только на необратимых швах.
- **P4. Механический триггер, не самоотчёт.** Риск/scope-расхождение считаются
  из журнала, а не из заявленной агентом «уверенности». Самооценке агента не
  доверяем как управляющему сигналу.
- **P5. События — источник истины; проекции одноразовы и безвластны.**
  `decisions.jsonl` — event log. Любой наблюдатель/дашборд — проекция:
  перестраиваема, не в критическом пути хука, fail-open.
- **P6. Enforcement-данные не редактируются подведомственным агентом.** Контракт,
  approval-штампы, `.gigacode/**` — в gated/штамп-тире (как `gate_spec_bootstrap`
  и self-protect), иначе гейт — театр.
- **P7. У всего опционального есть фолбэк.** Нет Serena → `rg`. Нет дашборда →
  CLI-дайджест + журнал. Нет Python → `run-hook.cjs` fail-open.

## 3. Таксономия контрол-поинтов (общий словарь)

| Знак | Тип | Останавливает? | Обход | Ловит |
|---|---|---|---|---|
| 🤖 | **автогейт** | по решению (`block`/`ask`) | структурно нет | ошибку + дрейф |
| 👁 | **checkpoint** | да, мягко | агент мог бы проскочить | честную ошибку |
| 🔒 | **human-gate** | да, жёстко (штамп/`ask`/PR-апрув) | структурно нет | ошибку + дрейф |
| ⚡ | **гибрид** | автогейт → `ask` материализует checkpoint в рантайме | — | overshoot |
| 📋 | **digest/notify** | нет (async) | — | дрейф направления на гранулярности стадии |
| 📡 | **проекция журнала** | нет (поток) | — | разбегание внутри стадии |
| ⏸ | **прерываемость** | по воле человека | — | что угодно, в любой момент |

Ключевое различение: **barrier ≠ наблюдаемость.** Больше барьеров = больше
ожидания. «Прозрачный контроль» достигается 📡 + 📋 + ⏸, а 🔒 тратятся только на
необратимое.

## 4. Целевой флоу

```
╔══════════════════════════════════════════════════════════════════╗
║ 📡 ПРОЕКЦИЯ ЖУРНАЛА   — включается на шаге 0, работает до конца    ║
║ ⏸ ПРЕРЫВАЕМОСТЬ       — активна на всех не-🔒 участках            ║
║ 🤖 gate_stage_order   — переход на стадию N+1 заблокирован, пока   ║
║    нет confirmation-артефакта стадии N. Любое «провалидируй» —     ║
║    это ЖЁСТКАЯ остановка, а не прозаическая просьба: следующая     ║
║    запись блокируется, агент НЕ может побежать дальше. Не под      ║
║    stop-budget (его нельзя «пересидеть»).                          ║
╚══════════════════════════════════════════════════════════════════╝

0. ЗАПУСК  /develop-feature | /fix-bug
   🤖 preflight_check · 🤖 gate_context_inject

1. INTAKE  (главная сессия, диалог)
   агент заполняет intake.json {task_type, mode, slug, scope_intent,
        acceptance|repro, constraints, missing_inputs}
   🤖 gate_intake_complete — вопросы ДЕРИВИРУЮТСЯ из пустых required-полей,
        а не из «спроси, если неясно». Все поля есть → вопросов НЕТ.
   🔒 understanding-checkpoint — агент рестейтит задачу СВОИМИ словами +
        in/out scope + открытые вопросы; инженер подтверждает ПОНИМАНИЕ
        до траты explore-токенов (ловит misread рано)

2. EXPLORE / CONTEXT  (read-only, fan-out шардами по модулям)
   🤖 git_guard (даже в чтении рубит деструктив)
   → 📋 digest: что нашёл · scope-кандидат · неуверенности · намерение шага 3

3. CONTRACT  ←—— заморозка скоупа (ПОСЛЕ explore, ДО записи)
   агент драфтит contract.json {scope_globs, modules, tests}
   🔒 contract-stamp — контракт = enforcement-данные, агент не самоутверждает

4. PLAN  (OpenSpec change)
   🤖 gate_spec_structure · 🤖 openspec validate --strict
   🔒 plan-stamp (HMAC) — несёт human-readable summary ПОДХОДА; инженер
        подтверждает понимание ПЛАНА (вторая половина «понял ли агент»)
   ⚠ fix-bug: шаги 3+4 схлопываются в ОДИН штамп (не дёргать дважды)

5. IMPLEMENT  (coder) ——————————————┐ петля под iteration-budget · ⏸
   на каждую правку:
     🤖 git_guard (деструктив/protected/self-protect/ветка → block)
     ⚡ gate_scope_guard: чужой модуль/.gigacode → block · overshoot → ask
     🤖 gate_lint (block) · 🤖 gate_clean_code (advisory)
   → 📋 digest после прохода ТОЛЬКО при scope-расхождении/ненулевом risk
                                      │
6. VERIFY / TESTS  ←———————————————————┘ loop-back при фейле
   🤖 gate_build · 🤖 gate_test (фейл → фидбэк coder'у)
   verifier → verdict.json {pass/fail, механические risk-поля}
   → 📋 digest вердикта ──CONFIDENCE-GATED (триггер механический)──┐
        risk низкий  → notify, петля идёт                          │
        risk высокий → 📋→🔒 авто-барьер ДО петли ре-деривации     │
   🤖 gate_iteration_guard — бюджет исчерпан → эскалация к человеку

7. DELIVERY  (Stop)
   🤖 validate_output: diff ⊆ contract.scope? · тесты заявлены/прогнаны?
                       manifest ⇿ диск? · validate --strict?
   👁 ревью pr-summary.md

8. COMMIT / PUSH / MERGE
   🔒 commit (settings ask) · 🔒 push (force deny)
   🔒 merge в protected — git_guard hard-block агенту → PR-апрув в Bitbucket

   ┌─ CHANGE-REQUEST (только через контракт, никаких out-of-band) ─┐
   │  те же модули → 🔒 amend-stamp дельты → переиграть 5–7        │
   │  новые модули → 🔒 новый контракт → с шага 3                  │
   └───────────────────────────────────────────────────────────────┘
```

Главный сдвиг против сегодняшнего репо: «confirm scope» и «re-derive» сейчас —
**прозаические** checkpoints. Контракт превращает первый в 🔒, машиночитаемый
вердикт — второй в детерминированную петлю под бюджетом. Жёсткими становятся ровно
те два места, где цена ошибки максимальна.

Второй сдвиг (по текущей боли intake/stage): вопросы агента перестают быть
случайными — они **деривируются** из пустых required-полей `intake.json`, а не из
наличия инструкции «спроси, если неясно» (ADR-7, P4). Появляются два явных
understanding-чекпоинта (понимание **задачи** на intake, понимание **подхода** на
plan-stamp), которых раньше не было. И стадии перестают «протекать»:
`gate_stage_order` делает каждое «провалидируй» **жёсткой остановкой** —
следующая запись блокируется, пока нет confirmation-артефакта, поэтому агент
физически не может попросить валидацию и побежать дальше (ADR-8).

## 5. Решения (ADR)

Формат: Контекст → Решение → Обоснование → **Contested** (где спорю сам с собой) →
Work items.

### ADR-1. Fan-out на чтение

- **Контекст:** лимит контекста (130K) бьёт по фазе разведки.
- **Решение:** `explore`/`repo-context` шардятся на несколько read-only
  субагентов по непересекающимся модулям. Только append в journal.
- **Обоснование:** записи нет → нет гонок за общим мутабельным состоянием
  (P5). Чистый выигрыш по контексту.
- **Contested:** требует, чтобы каждая запись журнала несла `session/feature/agent`
  (см. ADR-6, WI-2), иначе проекция не разведёт линии. Зависит от того, отдаёт ли
  форк параллельный диспатч субагентов (WI-V1).
- **Work items:** WI-9.

### ADR-2. Машиночитаемый контракт скоупа

- **Контекст:** scope сейчас — проза («confirm scope», «allowed write roots» в
  промпте). Не проверяется, не enforce'ится.
- **Решение:** `contract.json` рядом с `manifest.json`:
  `{scope_globs, modules, tests, status}`. Авторится **после** explore и **до**
  записи. Новый `gate_scope_guard` (PreToolUse WriteFile/Edit):
  - чужой модуль / `.gigacode/**` → `block`;
  - overshoot в смежное (DI, фикстура, миграция) → `ask` (журналит расширение);
  - path/dir-globs enforce'ятся дёшево (`_lib.matches_globs` + `_norm`).
  Заявленные тесты/символы — **advisory** на входе + **Stop-проверка**
  `diff ⊆ scope` и покрытие (символьный уровень в PreToolUse не проверить).
- **Обоснование:** P1. Несёт ADR-1 (модули = границы шардов) и ADR-3 (ключ
  бюджета). Change-request входит только через контракт (P6).
- **Contested:** контракт ДО explore был бы неверен → война с правильными
  правками; поэтому строго post-explore. Overshoot — `ask`, не `block`, иначе
  агент стопорится на корректном. «С начала на каждый change-request» дорого
  (P2, усталость) → расщепляем: amend (те же модули, дешёво) vs restart (новые
  модули). Сам контракт — в штамп-тире (P6), иначе агент расширяет скоуп при
  каждом упоре.
- **Аналогия:** per-task scoped lockfile + agent-enforced CODEOWNERS.
- **Work items:** WI-7, WI-8.

### ADR-3. Loop-agents как gate-governed петли

- **Контекст:** уже есть один loop-примитив — `stop_block_budget` (ограниченная
  петля с governor'ом; dev-версия закалена против «пересиживания» бюджета).
  verifier'ский `re-derive` существует прозой, без автоматизации и счётчика.
- **Решение:**
  1. **verdict.json** — verifier пишет машиночитаемый вердикт (`pass/fail` +
     механические risk-поля), петля ветвится по нему, не по парсингу прозы.
  2. **gate_iteration_guard** — обобщение `stop_block_budget` на любую стадию;
     ключ `stage:<feature>:<stage>` в `router-state.json`; эскалация к человеку
     при исчерпании.
  3. **test-fix loop** (тесты → фейлы → coder → перепрогон) ложится сюда, но
     **заблокирован** пустым `test.command` (WI-1).
- **Обоснование:** P2. Не тащим автономный loop-фреймворк — обобщаем governor,
  который уже построен.
- **Contested:** безлимитная петля = магия. Инвариант: счётчик + вердикт-артефакт
  + принудительная эскалация. Нет иначе.
- **Work items:** WI-10, WI-11, WI-1 (предусловие).

### ADR-4. Мульти-агентность: разделять чтения и записи

- **Контекст:** диспатч сегодня последовательный; параллелизма нет.
- **Решение:**
  - **Чтение** — fan-out безопасен (ADR-1).
  - **Запись** — параллельные писатели **запрещены**. Статусная машина
    (`scoping→draft→confirmed→complete`) последовательна; `router-state.json`
    last-write-wins; `manifest.json` — одно поле статуса; бюджет ключуется
    `stop:<session_id>`, не по агенту. Параллельная запись гонит гонку.
- **Обоснование:** P5 + детерминизм. Инфра сегодня не concurrency-safe.
- **Contested:** если параллельные писатели когда-нибудь нужны — отдельная
  инфра-работа: партиционирование state по агенту + файловые локи + agent-id в
  ключе бюджета. Не в этой итерации.
- **Платформа:** вероятно один уровень субагентов (main → subagent, без
  вложенности) → вся оркестрация петель живёт в главной сессии + гейтах, не
  внутри агента (WI-V2). И write-capable loop-агент обязан наследовать PreToolUse
  `git_guard` (WI-V3), иначе дыра множится с итерациями.

### ADR-5. Слой наблюдаемости (digest / projection / interrupt / confidence-gate)

- **Контекст:** «по CLI непонятно, что делает агент»; «сидеть и ждать» — не про
  этот флоу.
- **Решение:**
  - **📋 digest на шве субагента** — async notify, не barrier. Два блока:
    *механический* (проекция из `decisions.jsonl`: scope-diff, решения гейтов,
    цена) **рядом** с *нарративным* (намерение/неуверенность от агента);
    проекция подсвечивает расхождение. Человек может промотировать digest → 🔒.
  - **📡 projection** — всегда-вкл человекочитаемый вид журнала (стадия, бюджет,
    последние решения, scope-адгезия).
  - **⏸ interruptibility** — вмешательство между барьерами без планового стопа.
  - **confidence-gated барьер** на вердикте: единственный новый барьер; рутина —
    notify, в 🔒 промотируется только рискованное.
- **Обоснование:** P3. Барьеры не добавляем (они на необратимом) — добавляем
  поток + перехват.
- **Contested:** триггер промоушена — **механический** из журнала (размер
  scope-diff, число gaps/open-questions, сработал ли overshoot-`ask`, число
  модулей, остаток бюджета), не «агент сказал, что не уверен» (P4). Риск 📋-спама
  симметричен «стене ask» — дроссель: digest только при scope-расхождении или
  ненулевом risk; чистый проход = одна строка в 📡.
- **Платформа:** детерминированный digest-на-шве требует `SubagentStop` (в
  settings разведён только `SubagentStart`). Нет события → фолбэк на финальное
  сообщение агента (soft, дрейфует). Проверить (WI-V4).
- **Work items:** WI-12, WI-13, WI-11 (verdict risk-поля).

### ADR-6. Observer как read-model (CQRS), не control plane

- **Контекст:** запрос на локальную лёгкую приложеньку для наблюдения за флоу.
- **Решение:** observer = проекция поверх event log. **read-only в v1.**
  - источник истины — **file-tail `decisions.jsonl`**; push на localhost —
    опциональная оптимизация латентности (async + таймаут + fail-open, как
    serena-hooks). Enforcement **никогда** не в критическом пути наблюдателя
    (P5). Бонус file-tail — бесплатный replay постфактум.
  - стек: **zero-dep Python** (`http.server` + SSE + один статический HTML).
    Без билд-чейна, тривиально ревьюится — критично для банка. Nuxt/VueFlow —
    позже и только для граф-вью, на тот же `/stream`.
  - **развязать data-API (`/stream`) от фронта** — фронты сменяемы, контракт
    стабилен.
  - живёт в `modules/_shared/` (один observer на все модули) → ещё один довод за
    WI-3 и единую схему журнала (WI-2).
  - рендерит swimlanes по `session/feature/agent` (нужен WI-2), таймлайн стадий,
    бюджет-гейдж, поток решений с reason'ами, scope-адгезию (зелёный/красный по
    файлам против `contract.json`), risk-сигналы вердикта, две панели digest'а.
- **Обоснование:** P5. Read-model безвластен → не может сломать детерминизм.
- **Contested:**
  - *scope creep в control plane* — соблазн «апрув здесь» силён; именно там
    вьюшка становится властью, подрывающей гейты. Если добавляем управление —
    действие втекает через **тот же штамп-spine** (пишет подписанный approval,
    гейт независимо валидирует): app предлагает — гейт располагает.
  - *дашборд как единственное окно* — CLI-дайджест остаётся фолбэком (P7). App
    — приятный вид; журнал + CLI — гарантированный.
  - *лайфсайкл процесса* — не авто-спавнить из хука (ещё одна поверхность
    управления процессом); документированный one-liner во втором терминале.
- **Банковская поверхность:** bind строго `127.0.0.1` (никогда `0.0.0.0`),
  ноль phone-home/телеметрии. Журнал несёт пути/диффы/тикеты → app наследует
  чувствительность. На shared-машине/Citrix даже loopback бывает виден другим
  локальным юзерам — отметить в модели угроз.
- **Work items:** WI-14, WI-15, WI-2 (предусловие), WI-3 (предусловие).

### ADR-7. Детерминированный intake + understanding-чекпоинт

- **Контекст (текущая боль):** на intake/repo-context агент задаёт вопросы
  **случайно** — срабатывает само наличие инструкции «спроси, если неясно», а не
  реальный пробел. И нет точки, где инженер видит, **верно ли агент понял задачу
  и план**. Required-поля intake уже перечислены прозой в `development-flow`
  SKILL (feature: goal/behavior/acceptance/constraints; bug:
  symptom/expected/actual/repro/env/severity) — но не объявлены и не проверяются.
- **Решение:**
  1. **`intake.json`** — объявленный артефакт с required-полями (см. §8.5).
  2. **`gate_intake_complete`** (UserPromptSubmit / pre-explore) —
     детерминированно считает пустые required-поля и **возвращает ровно их как
     вопросы**. Все поля заполнены → вопросов нет. Это обобщение существующего
     `preflight_check` (он уже так делает для prompt'а) на весь intake-артефакт.
     Инструкция в скилле меняется с «спроси, если неясно» (vibes) на «заполни
     `intake.json`; чего не хватает — скажет гейт».
  3. **understanding-checkpoint (🔒)** — агент рестейтит задачу **своими словами**
     + in/out scope + открытые вопросы; инженер подтверждает **понимание** до
     explore. Вторая половина — понимание **подхода** — подтверждается на
     plan-stamp (human-readable summary плана). Это «дайджест на барьере»: тот же
     механический+нарративный формат (ADR-5), но синхронный.
- **Обоснование:** P4 (вопрос = функция от пустого поля, не от настроения
  модели) + P1 (декларация + гейт) + P3 (understanding-checkpoint стоит на
  реально необратимом — неверно понятая задача тратит весь прогон).
- **Contested:** понимание **задачи** дёшево и идёт **до** explore (ловит misread
  рано); понимание **подхода** требует explore → на plan-stamp, не раньше. Не
  плодить третий отдельный барьер — understanding встроен в уже существующие
  точки (intake + plan-stamp), а не добавлен сбоку.
- **Work items:** WI-20, WI-21.

### ADR-8. Энфорсмент стадий и непробиваемые остановки

- **Контекст (текущая боль):** агент часто просит провалидировать артефакты — и
  **тут же бежит дальше** в том же ходе. «Остановка» косметическая. Статусная
  машина (`scoping→draft→confirmed→complete`) есть, но как предусловие переходов
  **не enforce'ится**.
- **Решение:** **`gate_stage_order`** (PreToolUse, fail-closed) — переход на
  стадию N+1 разрешён только при наличии **confirmation-артефакта** стадии N
  (human-штамп там, где барьер 🔒; записанный ack — где мягкий 👁). Любая запись,
  относящаяся к следующей стадии, пока манифест в текущей стадии «awaiting
  validation», → `block`. Следствие: «попросить валидацию и побежать дальше»
  **структурно невозможно** — proceed-запись заблокирована. Сам запрос валидации
  не блокируется, блокируется только *продолжение*. Generalize'ит существующий
  `validate_run_output` (Stop) в enforced-переходы по всему циклу.
- **Обоснование:** P2 + P3. Явная остановка должна быть барьером, а не просьбой;
  иначе весь контроль протекает.
- **Contested:** **НЕ под `stop_block_budget`** — эти переходы человек-gated,
  «пересидеть» их нельзя (в отличие от Stop-деградации). Риск ложных блоков:
  стадии и их confirmation-артефакты должны быть перечислены явно в одном месте
  (stage-map в `router.config.json`), иначе гейт начнёт рубить легитимные
  внутристадийные записи. Эскейп-хетч (`disableAllHooks`) остаётся как у всех
  гейтов.
- **Work items:** WI-22.

## 8. Контракты данных (черновые схемы для старта)

### 8.1 `contract.json` (рядом с `manifest.json`)

```json
{
  "feature": "card-blocking",
  "status": "active",
  "stamp": "<hmac>",
  "scope_globs": ["src/cards/**", "src/common/dto/CardStatus.kt"],
  "modules": ["cards", "common-dto"],
  "tests": ["cards/CardBlockingServiceTest", "cards/CardStatusMapperTest"],
  "overshoots": []
}
```
`status`: `active | amended | superseded`. `overshoots` — журнал подтверждённых
`ask`-расширений (append). Правки `contract.json` — только через штамп (P6).

### 8.2 `verdict.json` (пишет verifier; ветвит петлю + питает confidence-gate)

```json
{
  "stage": "verify",
  "result": "fail",
  "risk": {
    "scope_diff_files": 7,
    "out_of_contract_files": 2,
    "open_questions": 3,
    "overshoot_asks": 1,
    "modules_touched": 2,
    "iteration_budget_left": 1
  },
  "findings": ["..."]
}
```
**Все поля `risk` — механические** (из журнала/диффа), не самооценка (P4). Порог
по ним промотирует digest → 🔒 (ADR-5).

### 8.3 `decisions.jsonl` — добавить идентификаторы линий (WI-2)

```json
{"kind":"gate","session_id":"s-42","feature":"card-blocking","agent":"coder",
 "event":"PreToolUse","tool":"Edit","gate":"gate_scope_guard",
 "decision":"ask","reason":"overshoot: src/common/...","ts":"2026-06-23T..."}
```
`session_id + feature + agent` обязательны в каждой записи → swimlanes,
per-agent бюджет, разведение fan-out.

### 8.4 `/stream` (SSE; источник — file-tail журнала)

```
event: decision
data: {"session_id":"s-42","feature":"card-blocking","agent":"coder",
       "stage":"implement","gate":"gate_lint","decision":"block",
       "reason":"...","budget_left":1,"ts":"..."}
```
Контракт стабилен; фронт (Python-HTML сейчас, VueFlow позже) — сменяем (ADR-6).

### 8.5 `intake.json` (объявленный intake; питает `gate_intake_complete`)

```json
{
  "task_type": "feature",
  "mode": "plan-only",
  "slug": "card-blocking",
  "scope_intent": "заблокировать карту по событию из anti-fraud",
  "acceptance": ["статус CARD меняется на BLOCKED", "событие в DLQ при сбое"],
  "repro": null,
  "constraints": ["без изменения публичного API CardController"],
  "missing_inputs": [],
  "understanding_ack": false
}
```
Required-набор зависит от `task_type` (feature: `scope_intent`/`acceptance`/
`constraints`; bug: `repro`/expected/actual/severity — см. `development-flow`
SKILL). `gate_intake_complete` возвращает **пустые required-поля как вопросы**;
`understanding_ack` ставит человек на understanding-checkpoint (ADR-7).

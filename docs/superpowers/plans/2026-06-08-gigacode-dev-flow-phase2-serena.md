# GigaCode Dev Flow — Phase 2 (Serena MCP) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Integrate Serena MCP as an optional semantic code-navigation provider in the GigaCode developer template, so agents are instructed and mechanically able to find existing code through Serena before proposing new implementations — directly addressing problem 1 ("agents ignore existing code").

**Architecture:** Serena is declared as an optional MCP server in `.gigacode/settings.json` with the `ide` context (recommended for Qwen3-Coder / GigaCode). A `.serena/` project scaffold is added to the template. Agent prompts and the `development-flow` skill are updated with a mandatory "search before create" rule using Serena's `find_symbol` tool. Serena's reminder hooks are wired through the existing `hook_router`-compatible structure in `settings.json`. All additions degrade gracefully when Serena is not installed.

**Tech Stack:** GigaCode (Qwen Code fork), Serena MCP (`serena-agent` via `uv`), Python 3, Bash + PowerShell smoke checks, Markdown/JSON/YAML config.

**Base branch:** `feature/dev-flow-enforcement` (worktree at `.worktrees/dev-flow-enforcement`). Phase 2 continues on this same branch.

**Reference spec:** `docs/superpowers/specs/2026-06-07-gigacode-dev-flow-enforcement-design.md`.

## Serena facts verified from official docs (do not guess, use these)

- **Install:** `uv tool install -p 3.13 serena-agent` (requires `uv`; `uv` install: `pip install uv` or see https://docs.astral.sh/uv/)
- **Init:** `serena init` — initialises the LSP backend; creates `~/.serena/serena_config.yml` globally.
- **Per-project file:** `.serena/project.yml` — created by `serena project create` in the project root.
- **MCP server command for Qwen/GigaCode:** `serena start-mcp-server --context ide --project-from-cwd`
  (Qwen3-Coder is listed as a supported client; `ide` context is recommended for CLI coding agents with no native Serena integration.)
- **Key tool name for finding existing code:** `find_symbol` (used as `mcp__serena__find_symbol` in hook matchers).
- **MCP server config JSON block** (goes into `settings.json` under `"mcpServers"`):
  ```json
  "serena": {
    "command": "serena",
    "args": ["start-mcp-server", "--context", "ide", "--project-from-cwd"]
  }
  ```
- **Reminder hooks** (help prevent agent drift; use `--client=claude-code` as the closest supported client):
  - `serena-hooks remind --client=claude-code` on `PreToolUse` (matcher `""`)
  - `serena-hooks auto-approve --client=claude-code` on `PreToolUse` (matcher `mcp__serena__*`)
  - `serena-hooks activate --client=claude-code` on `SessionStart`
  - `serena-hooks cleanup --client=claude-code` on `SessionEnd`
- **Graceful degradation:** if `serena` command is not found, the MCP server simply won't start; the template must document this and smoke checks must not require Serena.

```markdown
## Search Before Create

Before proposing any new function, class, module, or file, search for an existing
implementation using Serena's `find_symbol` tool:

```
mcp__serena__find_symbol(name_or_pattern="<SymbolName>")
```

If Serena is unavailable, use `rg` to search by symbol name across the repository:

```bash
rg -n "def <symbol_name>|class <SymbolName>|function <symbolName>" --type-add 'code:*.{py,ts,js,go,java,rs,rb,cs}' -t code
```

If a matching symbol is found:
1. Record its location in `docs/development/<task-slug>/context.md`.
2. Read the existing implementation before proposing changes.
3. Extend or adapt the existing code; do not write a duplicate.

If no match is found, record that in `context.md` and proceed with new implementation.
This rule applies in both plan-only and implement modes.
```
